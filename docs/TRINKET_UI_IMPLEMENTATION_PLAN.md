# Midnight Trinket UI / SIMC Integration Plan

## 1. 목표

한밤(MIDNIGHT) 시즌 기준 던전/레이드 장신구(TRINKET) 목록을 UI에서 보여주고, 사용자가 `trinket1`, `trinket2` 슬롯에 각각 장신구를 선택한 뒤 해당 조합으로 SimulationCraft 시뮬레이션을 실행할 수 있도록 한다.

이 문서의 범위는 다음 두 가지를 모두 포함한다.

- 사용자 UI/UX 설계
- 내부 데이터 모델 및 SIMC 프로필 생성 반영 계획

## 2. 요구사항 정리

### 2.1 사용자 기능 요구사항

- 한밤에서 드랍되는 던전/레이드 장신구 목록을 애플리케이션 UI에서 확인할 수 있어야 한다.
- 장신구는 슬롯 2개를 지원해야 한다.
- 사용자는 `trinket1`, `trinket2`를 각각 독립적으로 선택할 수 있어야 한다.
- 선택한 장신구로 SIMC를 실행할 수 있어야 한다.
- 기존 애드온 입력에서 파싱된 캐릭터 장비 정보는 유지하되, 사용자가 원할 때 장신구만 수동 override 할 수 있어야 한다.

### 2.2 내부 기술 요구사항

- SimulationCraft 프로필 생성 시 장신구 슬롯 override를 반영할 수 있어야 한다.
- 장신구 목록은 구조화된 데이터로 관리해야 한다.
- 중복 착용 가능 여부(`Unique-Equipped` 등)를 검증할 수 있어야 한다.
- 향후 시즌 교체 시 목록 데이터 교체 비용이 낮아야 한다.
- 현재 번들된 `simc.exe`에서 아이템을 실제로 해석할 수 없는 경우 사용자에게 실패 원인을 설명할 수 있어야 한다.

### 2.3 착수 전 확정 필요 항목

다음 항목은 구현 시작 전에 우선 합의하는 것을 권장한다.

- 한밤 장신구 목록의 기준 데이터 소스
- 각 장신구의 `itemId`, `unique` 여부 검증 기준
- `simc.exe` 미인식 아이템에 대한 UX 정책
- 난이도/아이템레벨을 1차에 포함할지 여부
- 장신구 UI의 정확한 배치 위치

## 3. 권장 구현 범위

### 3.1 1차 구현 범위

- 한밤 던전/레이드 장신구 목록 정적 데이터 추가
- 장신구 선택 UI 추가
- `trinket1`, `trinket2` 별 개별 선택 지원
- 선택값을 SIMC 프로필 생성 시 반영
- 입력값 검증 실패 시 사용자 메시지 표시
- override off 상태에서 기존 애드온 장신구 사용 흐름을 그대로 유지
- 현재 애드온 장신구와 수동 선택 장신구를 화면에서 명확히 구분

### 3.2 2차 확장 범위

- 난이도별 아이템 레벨 프리셋 제공

### 3.3 3차 확장 범위

- 향후 한밤 시즌2 이상 데이터 교체 자동화

## 4. 제안 UX

### 4.1 UI 배치

기존 캐릭터 입력/시뮬레이션 설정 흐름을 해치지 않기 위해 다음 두 방식 중 하나를 권장한다.

- 옵션 A: `CharInputPanel` 하단 또는 장비 관련 영역에 "장신구 선택" 그룹박스 추가
- 옵션 B: 시뮬레이션 옵션 패널에 "장비 Override" 섹션 추가 후 그 내부에 장신구 컨트롤 배치

권장안은 옵션 A이다. 사용자가 애드온 입력 결과와 장비 변경 UI를 한 화면에서 이해하기 쉽다.

### 4.2 UI 구성 요소

각 슬롯(`trinket1`, `trinket2`)마다 아래 컨트롤을 둔다.

- 체크박스: `수동 선택 사용`
- 콤보박스: 장신구 목록
- 콤보박스 또는 읽기전용 텍스트: 출처 표시
- 콤보박스: 난이도/아이템레벨 프리셋

추가 공통 컨트롤:

- 필터 체크박스 또는 탭: `던전`, `레이드`, `전체`
- 정보 텍스트: 선택 장신구 출처, 고유 착용 여부, 사용 효과 여부
- 검증 메시지 영역: 중복 착용 불가, 데이터 누락, SIMC 아이템 식별 불가 등

### 4.3 UX 동작

1. 기본 상태에서는 기존 애드온 파싱 장신구를 그대로 사용한다.
2. 사용자가 `수동 선택 사용`을 켜면 해당 슬롯만 override 된다.
3. 사용자가 장신구를 고르면 내부 선택 모델이 갱신된다.
4. 시뮬레이션 실행 시 override 값이 있으면 해당 슬롯만 수동값으로 프로필을 생성한다.
5. 두 슬롯 모두 override 시 두 장신구를 모두 반영한다.
6. 같은 장신구를 2개 선택했을 때 금지 아이템이면 시뮬레이션 실행 전에 막는다.

### 4.4 권장 UX 정책

- override를 켜지 않은 슬롯은 현재 애드온에서 파싱된 장신구명을 읽기전용으로 표시한다.
- 사용자가 수동 선택을 켜면 실제 시뮬레이션에 수동 선택이 적용된다는 점을 시각적으로 드러낸다.
- `itemId`가 없는 항목은 1차 구현에서 사용자 목록에 노출하지 않는다.
- 장신구 목록 정렬 기준은 "출처 + 이름" 또는 "이름 + 출처" 중 하나로 고정한다.

## 5. 데이터 모델 설계

### 5.1 장신구 마스터 데이터

장신구 목록은 문자열 배열이 아니라 구조체 기반으로 관리해야 한다.

예시 구조:

```cpp
struct TrinketOption
{
    CString displayName;
    CString sourceType;      // "dungeon" | "raid"
    CString sourceName;      // instance name
    CString bossName;        // optional
    int itemId = 0;
    CString simcName;        // optional fallback name
    CString bonusIdList;     // e.g. "12345/67890"
    int defaultItemLevel = 0;
    bool uniqueEquipped = false;
    bool onUse = false;
    CString roleHint;        // optional
    bool enabled = true;     // unsupported or incomplete entries can be hidden
};
```

권장 저장 위치:

- 1차: C++ 정적 테이블
- 2차: 외부 JSON 리소스

초기 구현에서는 C++ 정적 테이블을 우선 권장한다.

- 로딩 실패 케이스를 추가로 처리하지 않아도 된다
- 배포 구조 변경을 최소화할 수 있다
- 시즌 데이터가 안정되면 JSON 외부화로 확장 가능하다

### 5.2 사용자 선택 상태 모델

캐릭터 원본 데이터와 UI override 데이터를 분리한다.

예시 구조:

```cpp
struct TrinketOverrideSelection
{
    bool enabled = false;
    int itemId = 0;
    CString displayName;
    CString bonusIdList;
    int itemLevel = 0;
};
```

이를 `CharacterData` 또는 시뮬레이션 요청 DTO에 추가한다.

예시:

```cpp
TrinketOverrideSelection trinket1Override;
TrinketOverrideSelection trinket2Override;
```

핵심 원칙:

- 애드온 파싱 결과는 원본 유지
- UI 선택은 override 레이어로 별도 보관
- 프로필 생성 시점에 최종 merge

## 6. SIMC 프로필 생성 설계

### 6.1 반영 방식

SimulationCraft는 장비 슬롯별 선언을 지원하므로 최종 프로필 생성 시 장신구 라인을 아래 형태로 출력한다.

```text
trinket1=name,id=123456,bonus_id=111/222,ilevel=678
trinket2=name,id=654321,bonus_id=333/444,ilevel=678
```

중요한 점:

- 슬롯명은 `trinket1`, `trinket2`를 명시적으로 사용
- 가능하면 `item_id` 기준으로 생성
- `bonus_id`, `ilevel`은 데이터가 있을 때만 추가
- 이름은 가독성용으로 유지하되, 식별은 `id` 중심으로 간다

추가 원칙:

- 기존 장비 문자열 직렬화 포맷과 동일한 규칙을 유지한다.
- 기존 장신구 라인이 이미 생성되는 구조라면 append가 아니라 replace가 되어야 한다.
- 최종 프로필 내 `trinket1`, `trinket2` 라인이 중복 출력되지 않아야 한다.

### 6.2 최종 병합 로직

프로필 생성 시 슬롯별 우선순위:

1. override enabled + 유효한 itemId 존재
2. 애드온 파싱 결과 장신구
3. 둘 다 없으면 해당 슬롯 미출력 또는 경고

### 6.3 검증 로직

프로필 생성 전 아래 검증을 수행한다.

- `enabled == true` 인데 `itemId == 0` 이면 오류
- 동일 장신구 2개 선택 시 `uniqueEquipped == true` 이면 오류
- `ilevel` 또는 `bonus_id` 프리셋이 없는 경우 기본값 적용 또는 경고
- 현재 `simc.exe`에서 미인식 가능성이 있는 아이템이면 경고 후보로 기록

### 6.4 권장 예외 처리 정책

- `trinket1`만 override 되면 `trinket2`는 기존 파싱 장비를 유지한다.
- `trinket2`만 override 되면 `trinket1`는 기존 파싱 장비를 유지한다.
- 두 슬롯 중 하나라도 필수 검증에 실패하면 전체 실행을 막고 수정 포인트를 안내한다.
- `simc.exe` 미인식 가능성은 1차 구현에서 경고로 처리하고, 차단 정책은 2차에서 검토한다.

## 9. 데이터 소스 전략

### 9.1 필수 데이터 항목

각 장신구마다 최소한 아래 값이 필요하다.

- 표시 이름
- 출처 구분(던전/레이드)
- 인스턴스 이름
- item ID
- 기본 item level 또는 난이도 프리셋 정보
- 중복 착용 가능 여부

### 9.2 권장 수집 기준

가능하면 다음 우선순위로 수집한다.

1. 게임/DB 기반 item ID 확인 가능한 신뢰 가능한 출처
2. SimulationCraft에서 해석 가능한 아이템 기준 확인
3. 필요 시 내부 검증용 샘플 프로필로 실제 resolve 확인

실무 권장:

- 1차 데이터셋은 curated 수동 목록으로 관리한다.
- 자동 크롤링이나 외부 API 연동은 초기 범위에서 제외한다.
- 데이터 원본 출처와 확인 날짜를 별도 문서에 기록한다.

### 9.3 리스크

- 한밤 초기 시즌 데이터는 변경 가능성이 있다
- `simc.exe` 번들 버전이 최신 데이터베이스를 반영하지 못할 수 있다
- 사이트별 명칭과 실제 item ID가 다를 수 있다

대응:

- 데이터 소스를 코드 주석 또는 별도 문서에 기록
- 장신구 데이터 정의를 한 파일에 집중
- 장신구 인식 실패 시 사용자 경고 추가

## 10. 예외 및 정책

### 10.1 동일 장신구 2개 착용

기본 정책:

- 동일 장신구 2개를 선택할 수는 있게 하되
- `uniqueEquipped == true` 이면 실행 전에 차단

이유:

- UI 단계에서 무조건 막으면 예외 아이템 대응이 어렵다
- 검증 단계 차단이 유지보수에 더 유리하다

### 10.2 애드온 파싱 데이터와 충돌

정책:

- 장신구 슬롯만 override
- 나머지 장비는 기존 파싱 결과 유지
- override off 시 원상 복귀

### 10.3 장신구 데이터 누락

정책:

- 사용자 목록에는 `itemId` 없는 항목을 기본적으로 노출하지 않는다.
- 개발 로그에서만 누락 데이터를 추적한다.

## 11. 예상 구현 순서

가장 안전한 작업 순서는 아래와 같다.

1. 장신구 데이터 구조 추가
2. 내부 override 모델 추가
3. 프로필 생성 로직 확장
4. UI 연결
5. 검증/에러 처리
6. 빌드 및 smoke test

이 순서를 권장하는 이유는, UI보다 먼저 내부 모델과 프로필 생성 경로를 안정화하면 디버깅 범위를 줄일 수 있기 때문이다.

## 12. 예상 리스크와 완화책

### 리스크 1. `simc.exe`가 최신 한밤 장신구를 인식하지 못함

완화:

- 아이템 미인식 시 로그/경고 출력
- 장신구 데이터와 `simc.exe` 버전 매칭 체크

### 리스크 2. MFC UI resource 수정 범위가 커짐

완화:

- 기존 다이얼로그 레이아웃을 최대한 유지
- 신규 그룹박스와 콤보박스 중심의 최소 추가

### 리스크 3. 기존 장비 프로필 생성 로직 회귀

완화:

- override off 상태에서 기존 출력과 동일한지 비교
- 장신구 슬롯 관련 분기만 최소 추가

### 리스크 4. 시즌 교체 시 유지보수 비용 증가

완화:

- 장신구 마스터 데이터를 별도 파일로 분리
- 추후 JSON 외부화 가능하게 인터페이스 설계

### 리스크 5. 장신구 수가 많아 UI 사용성이 떨어짐

완화:

- 던전/레이드 필터를 기본 제공
- 출처와 이름을 함께 표시하는 렌더링 문자열 사용
- 필요 시 검색 가능한 콤보박스 또는 간단 그룹 분류를 2차에서 검토

### 12.1 구현 전 체크리스트

착수 전에 아래 항목을 먼저 확인하면 계획 품질이 높아진다.

- 한밤 장신구 최종 마스터 목록 확보
- `itemId`, `unique` 여부 검증 기준 확정
- `simc.exe` 현재 버전 및 한밤 아이템 해석 가능성 점검
- 실제 UI 삽입 위치와 dialog resource 공간 확인
- 기존 장비 override 기능 존재 여부 확인

## 13. 권장 후속 문서

실구현 착수 전에 아래 보조 문서를 추가로 만드는 것을 권장한다.

- `MidnightTrinketData.md`: 실제 장신구 리스트와 item ID 원본 정리
- `SimcProfileOverrideExamples.md`: 생성되어야 할 프로필 문자열 예시
- `UIControlMap.md`: 리소스 ID / 핸들러 연결표

## 14. 최종 제안

이번 작업은 "UI 표시"보다 "정확한 장신구 식별 데이터 관리"가 성공의 핵심이다.

따라서 구현은 아래 기준으로 진행하는 것이 가장 안전하다.

- 먼저 한밤 장신구 마스터 데이터부터 확정한다
- 다음으로 `trinket1`, `trinket2` override 모델과 프로필 생성 로직을 만든다
- 마지막에 MFC UI를 얹는다

이 접근을 따르면 UI는 단순한 선택기 역할에 집중하고, 실제 시뮬레이션 정확도는 중앙 데이터/프로필 생성 로직에서 일관되게 관리할 수 있다.

## 15. 재점검 결론

현재 계획은 구현 방향 자체는 적절하다. 다만 실제 개발 착수 전에는 아래 세 가지를 먼저 잠그는 것이 좋다.

- 장신구 데이터 기준 소스와 검증 정책
- `simc.exe` 미인식 시 사용자 경험 정책
- override on/off 상태를 보여주는 UI 방식

이 세 가지가 먼저 확정되면 중간에 되돌아오는 비용을 크게 줄일 수 있다.
